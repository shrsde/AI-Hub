STEP 2 OF 6
The terminal

The terminal is a text window where you type commands directly to your computer. It sounds intimidating but you only need about five commands to use Claude Code. Here's everything you need to know.

1
How to open it on a Mac

Press Cmd + Space to open Spotlight Search. Type Terminal and press Enter. A white or black window opens — that's it. You can also open the terminal directly inside Cursor: go to View → Terminal in the menu bar.

The terminal shows your username and a $ or % symbol. This is called the prompt — it means the computer is waiting for your instruction.

2
The five commands you actually need
CommandWhat it doesExample
cd"change directory" — moves you into a foldercd projects/myapp
lslists everything in the current folderls
mkdircreates a new foldermkdir my-project
pwdshows you exactly where you arepwd
claudeopens Claude Code in the current folderclaude

That's genuinely all you need. You'll use cd to navigate to your project, then claude to start working.

3
Navigating to your project — a worked example

Your project folder is probably in Documents or Desktop. Here's how to navigate there:

# Find out where you are right now
% pwd
/Users/yourname

# Move into your Documents folder
% cd Documents

# See what's inside
% ls
my-app/ notes/ projects/

# Move into your project folder
% cd my-app

# Now you're ready to open Claude Code here
% claude

Tip: you can drag a folder from Finder onto the terminal window and it automatically types the full path for you.

STEP 3 OF 6
Install Claude Code

With Node.js installed, Claude Code is a single command. Copy and paste each line exactly as written — don't change anything.

1
Install Claude Code globally

Open your terminal and run this command. The -g flag means "install globally" — so you can use it from any folder on your computer.

% npm install -g @anthropic-ai/claude-code

added 47 packages in 12s
...
✓ Claude Code installed successfully

If you see a permission error, run: sudo npm install -g @anthropic-ai/claude-code and enter your computer password when prompted.

2
Verify the install worked
% claude --version
@anthropic-ai/claude-code/1.x.x

If you see a version number — you're installed. If you see "command not found" — close the terminal completely, reopen it, and try again.

3
Install inside Cursor (optional but recommended)

Open Cursor. Press Cmd+Shift+X to open the Extensions panel. Search for Claude Code and click Install. This gives you a visual panel inside Cursor so you can see Claude working alongside your files.

Sign in with your Anthropic account when prompted. From this point on, you can use Claude Code either through the terminal or through Cursor's panel — whichever feels more comfortable.

STEP 5 OF 6
Set up your real project

Now let's do this properly. Every real project needs a CLAUDE.md file — the briefing document Claude reads at the start of every session. This is the single most important thing you can do to make Claude Code genuinely useful.

1
Navigate into your actual project
# Go to wherever your project lives
% cd ~/Documents/my-project

# Or if you're starting fresh, create it now
% mkdir ~/Documents/my-project
% cd ~/Documents/my-project
% git init
Initialized empty Git repository
2
Create your CLAUDE.md file

Create a file called CLAUDE.md in your project folder. Open it in any text editor — TextEdit on Mac works fine. Write in plain English. Here's a template:

# My Project Name

## What this is
One sentence describing what this app does and who it's for.

## What I'm building right now
The specific feature or thing I'm working on this session.

## Rules
- Always explain what you're going to do before doing it
- Never delete any file without asking me first
- When you write code, add a plain English comment above each section

## Out of scope right now
List anything you don't want Claude to build yet.

This file loads automatically every time you type claude in this folder. Claude immediately knows your project without you re-explaining it.

3
Verify Claude sees your briefing
% claude
> What do you know about this project?

Based on your CLAUDE.md, this project is...

If Claude describes your project accurately — you're set up correctly. Every session now starts with full context. This is the foundation everything else builds on.

Reference guide

Plain English
glossary

Every term you'll encounter when working with AI agents — explained without jargon. No technical background required.

Claude Code
An AI assistant that lives in your computer's command-line window and works directly inside your project files — reading, writing, and editing code on your behalf.
Like a contractor who works inside your actual office, not one who sends emails from across town.
CLAUDE.md
A plain text document you write once that tells the AI about your project — your stack, your rules, your current focus. It loads automatically at the start of every session.
The onboarding document you'd hand a new hire on day one.
Global vs project file
You have two instruction files. The global one applies to every project — your personal defaults. The project one only applies to that specific product.
Your personal work style is global. The brief for a specific client is project-level.
Context window
The amount of information the AI can hold in working memory during one session. The bigger the window, the more files and history it can track at once.
The size of someone's desk — bigger desk, more files spread out at once.
Skill
A folder with a plain-text instruction file inside. It teaches the AI a specific task — your exact process, your format, your rules. Claude loads it automatically when relevant.
A laminated instruction card above a workstation — your exact recipe, not a generic one.
Community skill
A pre-built skill someone else wrote and shared on GitHub. You download it and immediately have that capability — document generation, SEO analysis, slide creation.
A browser extension — install once, get the capability immediately.
MCP (Model Context Protocol)
The standard way to connect your AI to outside systems — your database, your files, GitHub, Slack. Once connected, the AI can look things up and take action instead of only knowing what you type.
Giving your contractor a keycard to the filing room and company database instead of them working from what you bring into the meeting.
MCP server
A specific connection to one external system. One for your database, one for your files, one for GitHub. Each is a separate tool the AI can pick up and use.
Each key on a keyring gives access to one room. More MCP servers = more rooms the AI can enter.
Subagent
A separate AI instance spawned to handle one specific job. Multiple subagents can work simultaneously on different parts of a task, then report back with results.
Dispatching specialists from your team to work in parallel, then reconvening to combine results.
Orchestrator
The main AI session that manages the overall task — breaks it into pieces, assigns work to specialists, and pulls everything back together.
The project manager who delegates to specialists, then assembles the final deliverable.
Always-on agent
An agent that runs automatically — triggered by an event (a file being saved, a PR being opened) or on a schedule (every morning at 8am). You don't trigger it. It just runs.
A security guard who patrols automatically — you don't have to ask them to check the doors.
Hook
A rule that fires automatically after every action the AI takes. For example: "after every file edit, run a quality check." The AI sees the result and fixes anything wrong before you look.
A spell-checker that runs automatically every time you finish typing — you never manually ask it to check.
Database
Where your app stores and retrieves information — user accounts, records, uploaded data. Everything your app needs to remember lives here.
The filing system behind the counter. The front of the shop is what users see; the database is everything behind it.
Deployment platform
The service that takes your code and makes it live on the internet. Every time you push an update, the platform rebuilds and publishes your app automatically.
The printer and delivery service — you write the document, it prints and ships to anyone who visits your URL.
Version control (GitHub)
Where your code lives online. Every change is saved with a description of what changed, so you can always go back to an earlier version or share code with others.
Google Docs version history for your entire codebase, plus a shared drive for your team.
API route
A URL on your server that your app can call to get data or trigger an action. When your app needs AI logic, it calls an API route on your own server, which then calls the AI.
An internal phone extension — dial the right number and a specific department handles your request.
Environment variable
A secret stored on your computer (not in your code files) that your app can read — like API keys and passwords. Keeps sensitive information out of your code.
Your bank PIN on a Post-it on your own desk rather than written inside a shared document.
Vibe coding
Building software by describing what you want in plain English and letting AI write the code — without needing to know how to code yourself.
Being the architect who draws the vision while the AI acts as the construction crew that builds it.
Fallback
What your app does when something fails — like the AI being temporarily unavailable. A good fallback queues the task for later or shows a "processing" message instead of breaking.
When your first supplier is out of stock, you automatically switch to your backup instead of cancelling the order.
Migration
A saved set of changes to your database structure — adding a new field, creating a new table. Migrations track changes so they can be applied or reversed precisely.
A documented change order for your filing system — recorded so you can undo it if needed.
Prompt injection
A security risk where hidden instructions inside a file try to trick the AI into doing something you didn't ask for — like overriding your rules or sending data elsewhere.
Someone hiding a note inside a document that says "ignore the contract and do this instead."
Stack block
A reusable section of your instruction file that describes a specific set of technologies you use. Write it once, paste it into any project that uses the same tools.
A template clause in a contract — written once carefully, reused whenever it applies.
.cursorrules
The equivalent of CLAUDE.md but for Cursor's built-in AI agent. Same idea — a plain text file at your project root that tells the AI your conventions and rules.
Same onboarding document, different contractor reading it.
Personal toolkit repo
A private GitHub repository where you store all your reusable AI assets — your global instruction file, stack blocks, skills, and agent definitions. Pull from it to start any new project instantly.
Your personal toolbox that you bring to every job site, instead of buying new tools each time.
Project configuration

Complete project
setup layers

Every new project is built from the same five layers. Each one adds a different kind of capability. Together they make every session start at full power.

The five layers

Layer 1 — context
CLAUDE.md instructions
Tells the AI who you are, what this project is, and how to behave. Two files: one global (applies everywhere) and one per project (specific to this product).
~/.claude/CLAUDE.md
/project/CLAUDE.md
Layer 2 — capability
Skills library
Plain-text instruction sets that teach the AI how to do specific tasks your way. Global skills apply to all projects. Project skills are specific to this product.
.claude/skills/
~/.claude/skills/
Layer 3 — roles
Agent definitions
Markdown files that define specialist roles the AI can adopt or spawn. The orchestrator calls on them for parallel or complex tasks.
.claude/agents/
~/.claude/agents/
Layer 4 — access
MCP server connections
What external systems the AI can reach — your database, your file system, GitHub, Slack, web search. Configured once in .mcp.json at your project root.
/project/.mcp.json
Layer 5 — extend
Community skills + marketplace plugins
Pre-built capabilities from the community — document creation, SEO, image generation, Figma integration. Install once, available everywhere.
github.com / marketplace

What your AI can reach

Once MCP servers are connected, your AI agents have direct access to your actual data and tools — not just what you paste into the chat.

filesystem
Your project files
The AI reads and writes your actual code. It knows what already exists, what patterns you use, and can make changes that fit — not generic code that might conflict.
database
Your live data
The AI queries your actual database tables. Instead of guessing at your schema, it looks it up. "What fields are we missing?" becomes a real question with a real answer.
github
Your code history
The AI reads your repositories, pull requests, and commit history. Useful for code review agents and automated summaries of what changed.
memory
Session scratchpad
A persistent notepad for the AI during long sessions. It remembers what it did in step 1 even when it's on step 8 — stops losing context mid-task.
web search
Live internet data
The AI can search the web to look up current documentation, APIs, library versions, or anything that may have changed since its training data.
deployment
Your live environment
Connected to Vercel, Netlify, or your host of choice. The AI can check deployment status, read logs, and surface errors from production.

Where your data lives

Database
Your app's memory. Stores user accounts, records, uploaded data, settings. Structured in tables. Queried through secure connections — never directly from the browser.
File storage
Where uploaded files live — images, documents, CSVs. Separate from your database. Usually an object storage service linked to your database by URL.
Deployment (Vercel etc.)
Your live app. Code runs here when users visit your URL. Environment variables (secrets) live here too — never in your code files.
GitHub
Your code history. Every version of everything you've ever built. Source of truth for your codebase — deployment pulls from here automatically on each push.
API services
Third-party tools your app calls — payment processors, AI models, email services, analytics. Accessed via keys stored as environment variables.
Local machine
Your computer. Where code is written and tested before being pushed to GitHub. Your .env.local file lives here — never committed, never shared.
How to work

The 6-phase
project process

A repeatable process for every project — from the first idea to a shipped feature. Click each phase to expand it.

1
Before you open the AI
5–10 minutes of thinking — before any tool is opened
▶
Define the problem in one sentence
Not the features — the problem being solved. Write it down. Everything the AI builds should answer to this sentence.
Why: Claude makes dozens of small decisions while building. A clear problem statement anchors every one of them.
Write the user flow step by step
What does a user actually do? Walk through every action in plain English. Not a design document — just the sequence.
Why: The AI can only build what you can describe. If you can't walk through it, the AI will invent its own version.
Decide what data you need to store
List the things your app needs to remember. Name them and what each contains. Don't worry about technical structure yet.
Why: Your database is the hardest thing to change later. Getting it right before building saves painful rework.
Set your scope boundary
Write a list of things explicitly out of scope for this build. This prevents the AI from enthusiastically adding features you didn't ask for.
→ When you can answer all four clearly, you're ready to open the AI. Not before.
2
Set up the layers
10–15 minutes — done once per project
▶
Write your project instruction file (CLAUDE.md)
Paste your stack block from your toolkit. Add your project brief on top — the problem, the user flow, the data you're storing, what's out of scope.
Copy your skills into .claude/skills/
Pull the relevant skill folders from your toolkit. These teach the AI how to do the specific tasks this project needs.
Copy your agents into .claude/agents/
Feature-planner and code-reviewer go in every project. Any domain-specific agents go in their respective projects.
Create your .mcp.json
Copy the starter template from your toolkit. Update the file path for this project. Add database connection details via environment variables — never hardcoded.
Verify everything loaded
Open the AI and ask "what do you know about this project?" It should summarise your instruction file correctly. If it can't — something isn't loading. Fix it now.
→ A verification check costs 2 minutes and prevents a broken foundation for the whole project.
3
Start building
The main phase — most of your time lives here
▶
Run the feature-planner agent first
Before writing a single line of code, ask the agent to plan the feature. It maps the database changes, components to build, and the order to build them.
→ Changing a plan costs nothing. Changing built code costs time and creates bugs.
Review the plan before building
Read the plan. Does it match your mental model? Are there missing steps? Make corrections in plain English now — before any code is written.
Build in small steps — one piece at a time
Tell the AI to build only the first item from the plan. When it's done and working, move to the second. Never ask it to build an entire feature in one go.
Test it yourself after each step
After each piece is built, click through it yourself. Does it do what you described? Check the empty state (no data), the error state, and the happy path.
Keep your instruction file current
As you build, your project evolves. When you add tables or finish major features, update the CLAUDE.md. An outdated brief is worse than none.
4
Use agents for big tasks
When a task has parallel workstreams or needs deep specialist work
▶
Know when to switch to agents
Two signals: the task has two or more clearly separate workstreams, or it would drain a single session's memory before it's done. Not every task needs this.
Write a clear parallel brief
Describe both workstreams explicitly and ask the AI to work on them simultaneously. Be specific about what each should produce and what format to return in.
Review, then direct the next step
When agents report back, treat their output as a briefing — not a final answer. Read it critically, then give one clear next instruction.
→ You stay in control of direction. Agents do the work; you make the decisions about what to do with it.
5
Before you ship
Non-negotiable gate before anything goes live
▶
Run the code-reviewer agent
Before merging any significant feature, ask the reviewer agent to check what was built. It explains everything in plain English so you understand what to fix — not just that something is wrong.
Walk through the user flow yourself
Open the preview and go through the exact flow you wrote in Phase 1, step by step. Every step. The bugs usually live in the transitions between steps.
Check the three states every screen must have
Real data view. Empty state (no data). Error state (something failed). If any of the three is missing or broken, fix it before shipping.
Confirm no secrets are in the code
Ask the AI to check all changed files for hardcoded passwords, API keys, or credentials. This takes 30 seconds and is the one mistake that's both easy to make and genuinely serious.
→ A secret committed to version control — even briefly — can be found by automated scanners instantly.
6
After the project
5 minutes — the phase most people skip, but shouldn't
▶
Update your instruction file with what changed
Add new tables, routes, or patterns. Change the "current focus" section. Make it accurate for the next session — not a snapshot from the start.
Update your toolkit if you refined anything
If you improved a skill or agent mid-project, update the file in your toolkit repo and commit it. That improvement now applies to every future project.
Write a one-paragraph "what I learned" note
What worked well? What did the AI get wrong repeatedly? What would you set up differently next time? Three minutes now, hours saved next project.
If you kept re-explaining something — make it a skill
If you found yourself explaining the same process or format more than twice, capture it as a skill now while the context is fresh. You stop paying the re-explanation tax on every future project.
→ Skills compound. Every refinement makes every future project better from day one.

What actually changes

Without agents
Your time equals the AI's speed. You're blocked waiting for each step. You're responsible for remembering every quality check. Problems get discovered after they're shipped. Each session starts from scratch re-explaining context.
With agents
Your time is decoupled from the AI's speed. You direct while it works. Quality checks run whether you remember to ask or not. Problems get caught before you look. Context loads automatically from your toolkit every session.

The role you play

Without agents, you are the dispatcher — give a task, wait, give the next one.
With agents, you become the director — define the work once, specialists run it, you come back to outcomes.

The biggest shift: you stop being the person who checks things and start being the person who only hears about things that actually need a decision.
How agents work

Four automation
patterns

Every always-on or parallel workflow fits one of these four patterns. Each one removes a different kind of back-and-forth from your day.

Pattern 1 — Parallel specialists
TRIGGERED BY YOU · RUNS SIMULTANEOUSLY · YOU WAIT ONCE
You give one instruction. Multiple specialist agents fan out and work on different parts of the same task at the same time. You wait for the longest task, not the sum of all tasks. One combined report comes back.
You say: "Work on these three things at the same time..."
→ Agent A researches the topic
→ Agent B reads your existing work
→ Agent C checks for problems
→ All three run simultaneously → one combined report → you review once

What skills are

Skills — teaching
the AI your way

A skill is a plain text instruction file that teaches the AI how to do a specific task exactly the way you want it done. Not a generic version — your version. Your format, your rules, your output.

Two types of skill

Global
Travels with you everywhere
~/.claude/skills/
Loaded automatically in every Claude Code session, regardless of which project you're in. These are universal capabilities you always want available — creating documents, writing SEO copy, generating slide decks.
  • Applies to every project you ever open
  • Lives on your machine, not in any repo
  • Mainly community skills — not project-specific
  • Set up once, available forever
Project
Specific to one product
[project]/.claude/skills/
Only loads when Claude Code is opened inside that project folder. These encode domain expertise specific to one product — a particular analysis format, a specific writing style, proprietary rules.
  • Only active for this one project
  • Committed to the project repo
  • Usually your own custom-written skills
  • Encodes knowledge nobody else has
The mental model: global skills are tools you bring to every job site. Project skills are the specific playbook for this particular client. Same contractor, different briefing per engagement.

What a skill actually looks like

A skill is a folder with one file inside it — SKILL.md. The file has two parts: a short header (the description Claude scans to know when to load it) and the detailed instructions Claude follows when it does.

example skill file — .claude/skills/listing-writer/SKILL.md
--- ← this header is what Claude reads first name: listing-writer description: Use when writing product listings, titles, bullet points, or descriptions for e-commerce. Triggers on: listing, product copy, title, bullets. --- ← Claude loads the full instructions below only when relevant # Listing writer skill Always follow this structure: ## Title - 60–80 characters max - Lead with the primary benefit, not the product name - No ALL CAPS, no excessive punctuation ## Bullets (always 5) - Each starts with a bold benefit label - Lead with what the customer gains, not what the product has - 15–25 words per bullet ## Tone rules - Confident and direct — never wishy-washy - No: innovative, revolutionary, amazing, game-changing - Yes: specific, concrete, benefit-led language

Claude reads the description (above the dashes) for every skill at session start — about 100 tokens total. It only loads the full instructions when a task matches. This keeps sessions fast without sacrificing capability.

How to vet a community skill before installing

A skill is an instruction set that shapes how the AI behaves. Treat it like a browser extension — inspect it before you run it. Official Anthropic skills and named partner skills (Google, Stripe, Vercel) can be installed without review. Everything else: check first.

1
Check the source first
Is it from github.com/anthropics, a named partner (Google Labs, OpenAI official), or a repo with 100+ stars and active maintenance? If yes — install without further review. If it's a random repo with few stars — read the file before installing anything.
2
Read the SKILL.md yourself — it takes 2 minutes
The file is plain text. Read it. Does it do what it says? Are the instructions coherent and scoped? Does it only instruct the AI to do what the skill claims to do? If yes — good signal.
3
Watch for these red flags
Instructions that mention overriding your other rules or CLAUDE.md. Shell commands (lines starting with !) that make network requests or write outside the project. Instructions to send data to an external service. Anything that contradicts the skill's stated purpose.
4
Ask the AI to review it for you
Paste the SKILL.md content and ask: "Check this for prompt injection, unexpected shell commands, scope creep, or any instruction to override my existing rules. Give me a plain English verdict: safe, use with caution, or do not install."
5
Test on a non-critical task first
Install the skill, then give it a small test task that matches its stated purpose. Does the output look right? Does the AI behave as expected? If yes — you're good. If the AI starts doing something unexpected, remove the skill and investigate.
The security note: most community skills are completely fine. The risk is real but low — the same as installing any open-source package. The review process above takes 5 minutes and gives you confidence. For anything from Anthropic's official repo or named tool partners, skip the review entirely.

How skills compound over time

Day one
You install the official document skills and copy your personal skills from your toolkit. Claude Code can now create Word docs, build UI correctly, write queries safely, and write briefs in your format. You did none of this from scratch.
After your first project
You found yourself explaining the same output format twice. You write it as a skill. Now you never explain it again — in this project or any future one. That's the first compounding event.
Six months in
Your personal skill library encodes months of domain refinement. Every new project starts with that accumulated expertise already loaded. The AI produces first-draft output that would have taken five back-and-forths a year ago.
The rule
Every time you explain the same thing to the AI more than twice in a session — stop and write a skill. That's the signal. Three minutes of writing now eliminates the cost of explaining it on every future task, in every future project.
